# Cloud Kitchen Order System — Design Specification

## Goals
- Support registered users and guest checkout.
- Menu management (items, prices, categories, party/catering flags).
- Kitchen staff can place orders for any customer (link by phone when available).
- Track order lifecycle and payment status (pending, paid, partially_paid, refunded, etc.).
- Record tips and manual payment entries (no card integration).
- Staff can apply discounts (percent or fixed).
- Support catering trays (small, medium, large) with defined pricing formula.
- Local deployment on a Mac; staff access via same Wi‑Fi.

---

## High-level architecture
- Single-server app (backend + DB) running on a Mac. Backend exposes a REST API and a simple web UI accessible on the local network.
- Postgres database (schema starts from provided kitch_* tables).
- Authentication: session-based or token (simple JWT) for registered users and staff.
- No external payment gateway integration — payments are recorded manually.

---

## Key business rules
- Registered users: sign up/login; orders linked to their account; can view order history.
- Guests: may place an order without an account; order stores guest phone/name.
- Staff phone-order: staff looks up cust by phone; if found, order is recorded under that user.
- Order tracking by order_id (public token or numeric id + verification).
- Payment statuses: pending, paid, partially_paid, refunded, cancelled. Tips stored separately.
- Discounts applied at order-level: either percent (e.g., 10%) or fixed amount. Staff can override totals.
- Catering pricing:
    - small = plate_price * 4 * 0.9
    - medium = plate_price * 6 * 0.9
    - large = plate_price * 12 * 0.9

## Data model (extensions + notes)
Keep and reuse provided tables (kitch_category, kitch_item_catalg, kitch_customer, kitch_status, kitch_payment, ...). Add/adjust tables below.

Note: use INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY for Postgres. Add FKs and constraints where noted.

- Users and Staff
```sql
CREATE TABLE kitch_user (
    user_id INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    cust_id INTEGER REFERENCES kitch_customer(cust_id), -- optional
    username VARCHAR(50) UNIQUE,        -- email or username for login
    password_hash VARCHAR(255),         -- bcrypt
    role VARCHAR(20) DEFAULT 'customer', -- 'customer','staff','admin'
    created_at TIMESTAMP DEFAULT current_timestamp
);
```

- Orders
```sql
CREATE TABLE kitch_order (
    order_id INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    order_ref VARCHAR(50) UNIQUE, -- optional public token for tracking
    cust_id INTEGER REFERENCES kitch_customer(cust_id), -- NULL for pure guests
    order_phone VARCHAR(12),      -- phone used for this order (guest or user)
    placed_by_user_id INTEGER REFERENCES kitch_user(user_id), -- staff or customer who placed
    order_date TIMESTAMP DEFAULT current_timestamp,
    subtotal NUMERIC(10,2) NOT NULL,
    discount_amount NUMERIC(10,2) DEFAULT 0,
    discount_type VARCHAR(10), -- 'percent' or 'fixed' or NULL
    tip_amount NUMERIC(10,2) DEFAULT 0,
    tax_amount NUMERIC(10,2) DEFAULT 0,
    total_amount NUMERIC(10,2) NOT NULL, -- final after discount/tax/tip
    payment_status VARCHAR(20) DEFAULT 'pending', -- pending, paid, partially_paid, refunded
    current_status_id INTEGER REFERENCES kitch_status(status_id),
    notes TEXT
);
```

- Order Items
```sql
CREATE TABLE kitch_order_item (
    order_item_id INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    order_id INTEGER REFERENCES kitch_order(order_id) ON DELETE CASCADE,
    kic_id INTEGER REFERENCES kitch_item_catalg(kic_id),
    name VARCHAR(100), -- denormalize name & price snapshot
    unit_price NUMERIC(10,2),
    quantity INTEGER DEFAULT 1,
    special_instructions TEXT,
    is_catering BOOLEAN DEFAULT FALSE,
    catering_size VARCHAR(10), -- 'small','medium','large' when is_catering TRUE
    line_total NUMERIC(10,2) -- unit_price * quantity or computed tray price
);
```

- Order status history (audit)
```sql
CREATE TABLE kitch_order_status_history (
    id INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    order_id INTEGER REFERENCES kitch_order(order_id),

```

- Optional: staff table for audit (or use kitch_user with role='staff').

## Important constraints & validations
- cust_phone_number format: exactly 10 digits (existing CHECK used).
- kic_price must be non-negative.
- Orders: subtotal = sum(line_total) before discounts/tips/taxes.
- Discounts:
    - If discount_type = 'percent', discount_amount represents percentage (0–100).
    - If discount_type = 'fixed', discount_amount is currency to subtract.
- Final total calculation performed server-side; store calculated fields to preserve history.
- When staff places order and phone matches existing kitch_customer.cust_phone_number, link cust_id to order.

## Catering pricing example (SQL formula)
Assume base plate price = kic_price from kitch_item_catalg.
- small: plate_price * 4 * 0.9
- medium: plate_price * 6 * 0.9
- large: plate_price * 12 * 0.9

Example SQL snippet to compute line_total for a catering item:
```sql
-- :size in ('small','medium','large')
SELECT
    k.kic_id,
    k.kic_name,
    CASE :size
        WHEN 'small' THEN k.kic_price * 4 * 0.9
        WHEN 'medium' THEN k.kic_price * 6 * 0.9
        WHEN 'large' THEN k.kic_price * 12 * 0.9
        ELSE k.kic_price
    END AS computed_price
FROM kitch_item_catalg k WHERE k.kic_id = :kic_id;
```

## API (suggested endpoints)
- Auth:
    - POST /api/register {username, phone, password}
    - POST /api/login {username, password} -> session token
- Customer:
    - GET /api/me/orders
    - GET /api/orders/{order_id} (only owner or staff)
- Orders:
    - POST /api/orders
        - payload: customer (cust_id or guest details), items [{kic_id, quantity, special_instructions, catering_size}], discount {type,value}, tip
        - server computes totals, creates order & items
    - POST /api/orders/phone-lookup {phone} -> returns cust_id if exists
    - GET /api/orders/track/{order_ref} -> public tracking (read-only)
    - PUT /api/orders/{order_id}/status {status_id, note} (staff)
- Payments:
    - POST /api/orders/{order_id}/payments {amount, payment_method, payment_status, tip_amount, notes, recorded_by}
    - GET /api/orders/{order_id}/payments
- Menu/Admin:
    - GET /api/menu
    - POST /api/menu (admin)
    - PUT /api/menu/{kic_id} (admin)
- Reports (admin):
    - GET /api/reports/sales?from=...&to=...
    - GET /api/reports/top-items?limit=5
    - GET /api/reports/top-customers?limit=5

## Reporting: sample SQL queries
- Total sales by day:
```sql
SELECT date_trunc('day', order_date) AS day,
             SUM(total_amount) AS total_sales,
             COUNT(*) AS orders_count
FROM kitch_order
WHERE order_date BETWEEN :start AND :end
GROUP BY day ORDER BY day;
```
- Total sales by item (period):
```sql
SELECT oi.kic_id, oi.name, SUM(oi.line_total) AS sales, SUM(oi.quantity) AS qty_sold
FROM kitch_order_item oi
JOIN kitch_order o ON oi.order_id = o.order_id
WHERE o.order_date BETWEEN :start AND :end
GROUP BY oi.kic_id, oi.name
ORDER BY sales DESC;
```
- Top 5 selling items:
```sql
... same as above ... LIMIT 5
```
- Top 5 users (by spend):
```sql
SELECT c.cust_id, c.cust_name, SUM(o.total_amount) AS total_spent
FROM kitch_order o
JOIN kitch_customer c ON o.cust_id = c.cust_id
GROUP BY c.cust_id, c.cust_name
ORDER BY total_spent DESC LIMIT 5;
```

## UI / UX flows (brief)
- Landing page: menu + login/register + guest checkout CTA.
- Cart: item qty, special_instructions per item, catering size selector when applicable.
- Checkout:
    - Guest: collect name + phone
    - Registered: use saved profile + phone
    - Staff mode: search by phone to attach existing customer
    - Show calculated totals, discount input (staff only), tip input.
    - Finalize -> create order; show order_ref for tracking.
- Order tracking: enter order_ref -> show status timeline.

## Deployment & local network
- Run backend (e.g., Flask/Express/Rails) on local Mac, bind to 0.0.0.0 so devices on same Wi‑Fi can access via machine IP and port.
- DB: local Postgres instance. Use role kitchen_user as provided.
- Recommend using docker-compose for reproducible local setup (Postgres + app).
- Try to develop using python as base language for backend.

---

## Security & operational notes
- Store password hashes (bcrypt). Use TLS on network if sensitive.
- Limit staff/admin endpoints to authenticated users with appropriate role.
- Validate phone numbers and enforce uniqueness for registered customers.
- Keep audit logs for status/payment changes.
- Data backups and simple migration path for schema changes.

---

## Testing & acceptance criteria
- Register/login flows for user and staff.
- Guest order creation and retrieval by order_id.
- Staff order creation linked to existing customer by phone.
- Correct computation of catering sizes and discounts (unit tests).
- Payment records reflect manual status changes and tips.
- Reports return correct aggregations for sample data.
