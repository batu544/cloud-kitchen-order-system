# Cloud Kitchen Order System ŌĆö Implementation Specification

## Overview
A Python/Flask REST API + PostgreSQL + Tailwind CSS web UI for managing a local cloud kitchen. Supports registered users, guest checkout, staff phone orders, catering pricing, manual payment tracking, and admin reporting. Deployed locally on a Mac; staff access via Wi-Fi.

## Key Business Rules

- **Registered users**: sign up/login; orders linked to their account; can view order history.
- **Guests**: place orders without an account; order stores guest phone/name.
- **Staff phone-order**: staff looks up customer by phone; if found, order is linked to that customer record.
- **Order tracking**: public tracking via `order_ref` (format: `ORD-YYYYMMDD-{hex}`).
- **Payment statuses**: `pending`, `paid`, `partially_paid`, `refunded`, `cancelled`. Tips stored separately.
- **Discounts**: order-level only; staff/admin can apply. Either percent (0ŌĆō100) or fixed amount.
- **Tax**: always 0.00 (no tax applied).
- **Catering pricing** (from `kic_price`):
  - small  = plate_price ├Ś 4 ├Ś 0.9
  - medium = plate_price ├Ś 6 ├Ś 0.9
  - large  = plate_price ├Ś 12 ├Ś 0.9
- **Password**: minimum 8 characters (validated server-side before any other check).
- **Phone**: exactly 10 digits; enforced as CHECK constraint and in application validators.

## Data Model

### Base tables (existing, unchanged)
`kitch_category`, `kitch_item_catalg`, `kitch_customer`, `kitch_status`, `kitch_payment`

Default statuses: Pending (1), Confirmed (2), Preparing (3), Delivered (3.5), Ready (4), Completed (5), Cancelled (6)

Default categories: Appetizers, Main Course, Desserts, Beverages, Catering

### kitch_user
```sql
CREATE TABLE kitch_user (
    user_id     INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    cust_id     INTEGER REFERENCES kitch_customer(cust_id),  -- optional link
    username    VARCHAR(50) UNIQUE NOT NULL,  -- email or username for login
    password_hash VARCHAR(255) NOT NULL,      -- bcrypt
    role        VARCHAR(20) DEFAULT 'customer',  -- 'customer','staff','admin'
    is_active   BOOLEAN DEFAULT TRUE,
    created_at  TIMESTAMP DEFAULT current_timestamp
);
```

### kitch_order
```sql
CREATE TABLE kitch_order (
    order_id         INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    order_ref        VARCHAR(50) UNIQUE,          -- public token: ORD-YYYYMMDD-{hex}
    cust_id          INTEGER REFERENCES kitch_customer(cust_id),  -- NULL for guests
    order_phone      VARCHAR(12) CHECK (order_phone ~ '^\d{10}$'),
    placed_by_user_id INTEGER REFERENCES kitch_user(user_id),
    order_date       TIMESTAMP DEFAULT current_timestamp,
    subtotal         NUMERIC(10,2) NOT NULL,
    discount_amount  NUMERIC(10,2) DEFAULT 0,
    discount_type    VARCHAR(10),   -- 'percent' | 'fixed' | NULL
    tip_amount       NUMERIC(10,2) DEFAULT 0,
    tax_amount       NUMERIC(10,2) DEFAULT 0,     -- always 0
    total_amount     NUMERIC(10,2) NOT NULL,       -- after discount + tip
    payment_status   VARCHAR(20) DEFAULT 'pending',
    current_status_id INTEGER REFERENCES kitch_status(status_id),
    notes            TEXT
);
```

### kitch_order_item
```sql
CREATE TABLE kitch_order_item (
    order_item_id       INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    order_id            INTEGER REFERENCES kitch_order(order_id) ON DELETE CASCADE,
    kic_id              INTEGER REFERENCES kitch_item_catalg(kic_id),
    name                VARCHAR(100),      -- denormalized price snapshot
    unit_price          NUMERIC(10,2),
    quantity            INTEGER DEFAULT 1,
    special_instructions TEXT,
    is_catering         BOOLEAN DEFAULT FALSE,
    catering_size       VARCHAR(10),       -- 'small' | 'medium' | 'large'
    line_total          NUMERIC(10,2),     -- computed; preserved for history
    updated_at          TIMESTAMP DEFAULT current_timestamp  -- for optimistic locking
);
```

### kitch_order_status_history

## Validations & Constraints

- `order_phone`: exactly 10 digits (DB CHECK + application validator)
- `kic_price`: non-negative
- `password`: minimum 8 characters (validated first, before any other registration check)
- `username`: minimum 3 characters
- `discount_type`: `'percent'` ŌåÆ discount_amount is 0ŌĆō100; `'fixed'` ŌåÆ subtract from subtotal (capped at subtotal)
- All totals calculated server-side; client-submitted totals are ignored
- Phone uniqueness enforced for registered customers (one user account per phone)
- Cannot delete the last item from an active order

## Business Logic ŌĆö Order Total Calculation

```
subtotal     = sum(item.line_total for all items)
discount     = subtotal * (discount_amount / 100)   if discount_type == 'percent'
             = min(discount_amount, subtotal)         if discount_type == 'fixed'
             = 0                                      otherwise
tax          = 0  (no tax)
total_amount = (subtotal - discount) + tip_amount
```

All fields stored at order creation to preserve history. Totals are recalculated when staff edits items or discount.

## Business Logic ŌĆö Auto Status Transition

When a payment with `payment_status = 'paid'` is recorded on an order whose `current_status` is "Delivered", the order is automatically transitioned to "Completed".

## Reporting Queries

### Sales by period
```sql
SELECT DATE_TRUNC(%s, order_date) AS period,
       SUM(total_amount) AS total_sales,
       COUNT(*) AS orders_count,
       AVG(total_amount) AS avg_order_value
FROM kitch_order
WHERE order_date BETWEEN %s AND %s
  AND payment_status != 'cancelled'
GROUP BY DATE_TRUNC(%s, order_date)
ORDER BY period;
```

### Sales by item
```sql
SELECT oi.kic_id, oi.name, SUM(oi.line_total) AS sales, SUM(oi.quantity) AS qty_sold
FROM kitch_order_item oi
JOIN kitch_order o ON oi.order_id = o.order_id
WHERE o.order_date BETWEEN %s AND %s
GROUP BY oi.kic_id, oi.name
ORDER BY sales DESC;
```

### Top customers by spend
```sql
SELECT c.cust_id, c.cust_name, SUM(o.total_amount) AS total_spent
FROM kitch_order o
JOIN kitch_customer c ON o.cust_id = c.cust_id
GROUP BY c.cust_id, c.cust_name
ORDER BY total_spent DESC LIMIT %s;
```

## UI / UX Flows

### Guest Checkout
1. Browse menu (no login required)
2. Add items to cart; select catering size if applicable; add special instructions
3. At checkout: enter name + phone
4. Server creates order; returns `order_ref` for tracking

### Registered User Checkout
1. Login ŌåÆ profile phone pre-filled at checkout
2. Same cart flow as guest

### Staff Phone Order
1. Enter customer phone in staff new-order page
2. System auto-links existing `kitch_customer` if phone matches
3. Staff selects items, optionally applies discount, records tip
4. Creates order ŌåÆ updates status as order progresses
5. Records manual payment with optional tip/override

### Order Tracking (Public)
Enter `order_ref` on the track page ŌåÆ displays status timeline from `kitch_order_status_history`.

## Deployment

```bash
# 1. Clone and activate virtual environment
source .venv/bin/activate
pip install -r requirements.txt

# 2. Configure environment
cp .env.example .env   # fill in DB credentials and JWT_SECRET_KEY

# 3. Create PostgreSQL database
createdb kitchen_db
psql kitchen_db -c "CREATE USER kitchen_user WITH PASSWORD '...';"
psql kitchen_db -c "GRANT ALL PRIVILEGES ON DATABASE kitchen_db TO kitchen_user;"

# 4. Run migrations
python -m src.database.migrate

# 5. Load demo data (admin/staff/customer users + sample menu)
python -m src.database.seed

# 6. Start server
python run.py
# Access: http://<mac-ip>:5000 (binds 0.0.0.0 for Wi-Fi access)
```

## Security

- **Passwords**: bcrypt hashed; minimum 8 characters enforced
- **Auth**: JWT Bearer tokens; 24-hour expiry (configurable)
- **Roles**: `customer` / `staff` / `admin`; enforced via `@require_role` decorator
- **Rate limits**: register 5/min, login 10/min
- **Security headers**: X-Content-Type-Options, X-Frame-Options, X-XSS-Protection, CSP, Referrer-Policy
- **SQL**: parameterized queries only (no f-strings in SQL)
- **Audit trail**: all status changes in `kitch_order_status_history`; all item edits in `kitch_order_edit_history`
- **Phone uniqueness**: one user account per phone number
